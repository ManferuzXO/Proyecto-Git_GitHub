01 / INTRO

¿Qué es Git y GitHub?

🌳
Git — Control de versiones
Git es un sistema de control de versiones distribuido creado por Linus Torvalds (2005). Permite rastrear cambios en el código, revertir errores y trabajar en paralelo sin pisarse entre colaboradores.
🐙
GitHub — Plataforma remota
GitHub es una plataforma web que aloja repositorios Git en la nube. Añade herramientas de colaboración: Pull Requests, Issues, Actions (CI/CD), Pages, y más.
📦
Repositorio
Un repositorio (repo) es la carpeta del proyecto que Git vigila. Contiene todos los archivos y el historial completo de cambios desde el inicio.
🔄
Commit
Un commit es una "foto" del estado del proyecto en un momento dado. Cada commit tiene un ID único (hash SHA-1), un mensaje, autor y fecha.
🌿
Rama (Branch)
Una rama es una línea de desarrollo independiente. Permite desarrollar features o corregir bugs sin afectar la rama principal (main/master).
🔁
Merge & Pull Request
El merge une dos ramas. En GitHub, un Pull Request es la solicitud formal para que tus cambios sean revisados y fusionados al proyecto principal.
Dato clave: Git funciona localmente en tu máquina — no necesitas internet para hacer commits. GitHub es el servidor remoto donde sincronizas con push y pull.
Dato curioso: GitHub guardó 21 TB de código en una mina en Noruega para que sobreviva 1,000 años en caso de un desastre global.

Initialized empty Git repository in /home/user/mi-proyecto/.git/

user@pc:~/mi-proyecto (main) $ git status
On branch main
Untracked files:

user@pc:~/mi-proyecto (main) $ git add README.md
# 4. Crear el commit
user@pc:~/mi-proyecto (main) $ git commit -m "docs: add README inicial"
[main (root-commit) a1b2c3d] docs: add README inicial

user@pc:~/mi-proyecto (main) $ git remote add origin git@github.com:usuario/mi-proyecto.git
user@pc:~/mi-proyecto (main) $ git push -u origin main
Branch 'main' set up to track remote branch 'main' from 'origin'.

04 / RAMAS

Trabajo con ramas

Las ramas permiten desarrollar en paralelo. El flujo estándar: crear rama → desarrollar → merge a main.

main ●───────────────────────────●──● ← HEAD
\ /
feature/ ●──────●──────●
● = commit ● = merge commit ← HEAD = posición actual
bash — ciclo completo de rama
# Crear y cambiar a la nueva rama
user@pc:~/proyecto (main) $ git switch -c feature/login
Switched to a new branch 'feature/login'
# Desarrollar y commitear
user@pc:~/proyecto (feature/login) $ git add src/login.py
user@pc:~/proyecto (feature/login) $ git commit -m "feat: implement login endpoint"
[feature/login 3d4e5f6] feat: implement login endpoint
# Subir la rama al remoto
user@pc:~/proyecto (feature/login) $ git push -u origin feature/login
# Volver a main y hacer merge
user@pc:~/proyecto (feature/login) $ git switch main
user@pc:~/proyecto (main) $ git merge feature/login --no-ff -m "merge: feature/login"
Merge made by the 'ort' strategy.
# Eliminar la rama local ya mergeada
user@pc:~/proyecto (main) $ git branch -d feature/login
Deleted branch feature/login (was 3d4e5f6).
Convención de nombres de ramas:
feature/nombre-feature — nuevas funcionalidades
fix/nombre-bug — corrección de errores
hotfix/nombre — corrección urgente en producción
docs/nombre — cambios en documentación
refactor/nombre — reestructuración de código
05 / COLABORACIÓN

GitHub: trabajo en equipo

Un Pull Request (PR) es la forma estándar de proponer cambios en GitHub. Permite revisión de código antes de hacer merge.

01
Crear una rama y hacer commits
Trabaja en tu feature en una rama separada, nunca directamente en main.
git switch -c feature/mi-feature git commit -m "feat: ..."
02
Subir la rama a GitHub
Sube tu rama al repositorio remoto para que sea visible en GitHub.
git push -u origin feature/mi-feature
03
Abrir el Pull Request en GitHub
En GitHub, ve a tu repo → "Compare & pull request". Escribe un título descriptivo y una descripción explicando qué cambiaste y por qué.
04
Revisión de código (Code Review)
Los colaboradores revisan el código, dejan comentarios y pueden pedir cambios. Responde a los comentarios y haz los ajustes necesarios con nuevos commits.
05
Merge del Pull Request
Una vez aprobado, se hace el merge a main. Se puede usar "Squash and merge" para limpiar el historial o "Merge commit" para conservarlo todo.
06
Actualizar tu local
Después del merge, actualiza tu rama main local.
git switch main git pull git branch -d feature/mi-feature

El fork es para contribuir a repositorios donde NO tienes acceso directo (proyectos de terceros o open source).

01
Hacer Fork del repositorio
En GitHub, presiona el botón "Fork" en el repositorio original. Esto crea una copia tuya en tu cuenta.
02
Clonar TU fork
git clone git@github.com:TU_USUARIO/repo-forkeado.git cd repo-forkeado
03
Agregar el remoto original (upstream)
git remote add upstream git@github.com:ORIGINAL/repo.git git remote -v
04
Desarrollar en una rama y hacer push
git switch -c fix/mi-corrección git push origin fix/mi-corrección
05
Abrir PR desde tu fork al original
En GitHub, en tu fork aparecerá un botón para "Compare & pull request" hacia el repo original.
06
Sincronizar con el upstream
git fetch upstream git switch main git merge upstream/main

Un conflicto ocurre cuando dos personas modificaron la misma línea del mismo archivo. Git no puede decidir solo cuál es la correcta.

bash — resolver conflicto de merge
user@pc:~/proyecto (main) $ git merge feature/login
CONFLICT (content): Merge conflict in src/app.py
Automatic merge failed; fix conflicts and then commit the result.
# El archivo tendrá marcadores de conflicto:
# <<<<<<< HEAD
# código de TU rama
# =======
# código de la otra rama
# >>>>>>> feature/login
# Edita el archivo, elimina los marcadores, deja el código correcto
# Luego:
user@pc:~/proyecto (main|MERGING) $ git add src/app.py
user@pc:~/proyecto (main|MERGING) $ git commit -m "merge: resolver conflicto en app.py"
[main 7g8h9i0] merge: resolver conflicto en app.py

SSH permite conectarte a GitHub sin escribir usuario/contraseña en cada push.

# 1. Generar clave SSH

Generating public/private ed25519 key pair.
Enter file in which to save the key: [Enter]

# 2. Copiar la clave pública
user@pc:~ $ cat ~/.ssh/id_ed25519.pub

# 3. Ve a GitHub → Settings → SSH Keys → New SSH Key
# Pega la clave pública ahí
# 4. Verificar conexión
user@pc:~ $ ssh -T git@github.com

01
Crear carpeta .github/workflows
En tu repositorio, crea la carpeta para los workflows de Actions.
mkdir -p .github/workflows
02
Crear archivo workflow.yml
Crea un archivo YAML con la configuración del workflow.
03
Definir triggers y jobs
Especifica cuándo ejecutar (push, pull request) y qué hacer (instalar dependencias, ejecutar tests).
04
Hacer commit y push
Sube el workflow a GitHub para activarlo.
git add .github/workflows/ git commit -m "ci: add GitHub Actions workflow" git push
yaml — ejemplo de workflow básico
# Archivo: .github/workflows/ci.yml
name: CI
on:
push:
branches: [ main ]
pull_request:
branches: [ main ]
jobs:
test:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v4
- name: Set up Python
uses: actions/setup-python@v4
with:
python-version: '3.9'
- name: Install dependencies
run: pip install -r requirements.txt
- name: Run tests
run: python -m pytest
¿Qué hace este workflow?
Se ejecuta en cada push/PR a main, instala Python, dependencias y corre tests automáticamente.
Beneficios: Detecta errores temprano, asegura calidad del código, automatiza despliegues.

07 / HISTORIA

¿Cómo nació todo esto?

Dos herramientas que cambiaron la forma de desarrollar software, nacidas en circunstancias muy distintas.

2005
🌳
Nace Git
Linus Torvalds, el creador del kernel de Linux, necesitaba una herramienta de control de versiones rápida y distribuida. La herramienta que usaba antes (BitKeeper) retiró su licencia gratuita, así que Torvalds decidió crear la suya propia. En solo 10 días escribió el primer prototipo de Git.
Linus Torvalds · Linux Foundation
2008
🐙
Nace GitHub
Tom Preston-Werner, Chris Wanstrath y PJ Hyett lanzaron GitHub como una plataforma para alojar repositorios Git en la nube. La idea era simple: hacer que colaborar en código fuera tan fácil como compartir un enlace. En su primer año ya tenía más de 46,000 repositorios públicos.
Preston-Werner · Wanstrath · Hyett
2018
💼
Microsoft adquiere GitHub
Microsoft compró GitHub por 7,500 millones de dólares. Muchos desarrolladores temieron que la plataforma cambiara, pero siguió siendo gratuita y de código abierto. Hoy GitHub tiene más de 100 millones de usuarios y es el repositorio de código más grande del mundo.
Microsoft · $7.5B
Hoy
🌍
El estándar global
Git es usado por prácticamente todos los equipos de desarrollo del mundo. GitHub aloja proyectos como Linux, React, Python, VS Code y millones más. Aprender Git no es opcional — es una habilidad fundamental para cualquier desarrollador.
+100M usuarios · +420M repositorios
Dato curioso: El nombre "Git" fue elegido por Torvalds con humor. En slang británico, "git" significa algo así como "idiota". Dijo: "Soy un egoísta y le pongo mi nombre a todos mis proyectos. Primero Linux, ahora Git."
